## Clarifications

### Session 2026-07-13

- Q: What is the source of truth for the target visual design? → A: The `Frontend/design_handoff 2/` folder (high-fidelity HTML/React prototype generated by Claude Design). The redesign recreates those designs pixel-accurately in the existing React stack; `Event Management System.dc.html` is the authoritative reference for markup/spacing/states, with `README.md`/`DESIGN-CONTEXT.md` for tokens and screen inventory.
- Q: Is 007 a pure re-skin, or does it include net-new screens/features and backend work? → A: 007 is the **full Redesign initiative** (grilled 2026-07-12; [ADR-007](../../docs/decisions/007-hubspot-custom-objects-registration.md) / [ADR-008](../../docs/decisions/008-standalone-events-event-first-nav.md) / [ADR-009](../../docs/decisions/009-redesign-ui-platform-theming-typography.md)): **UI redesign + HubSpot custom-objects migration**, including new content and backend work. Delivered in two phases — **Phase A (unblocked):** semantic token layer + 3 themes (incl. net-new Dark Aurora), self-hosted Manrope + Material Symbols, a11y field pickers, backend theme-preference endpoint. **Phase B (blocked on `X-REDESIGN-001` feasibility gates):** event-first navigation, standalone Events, registration-as-association. Phase B is gated on HubSpot freeing 2 custom-object slots.
- Q: How much of Phase B should this spec specify now? → A: **Fully specify both Phase A and Phase B in this one spec** (detailed acceptance criteria for theming/persistence, event-first navigation, standalone Events, and registration-as-association), rather than deferring Phase B detail to later per-surface specs. Phase B requirements remain gated on `X-REDESIGN-001` for *delivery*, but are documented in full here.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Staff experience a modernized, consistent UI (Priority: P1)

Event operations staff (front-of-house, catalog admins, account managers) use the EMS day-to-day for check-in, email, and catalog management. Today's UI is functional but has grown incrementally slice-by-slice; the redesign initiative brings it to a consistent, modern visual standard without breaking existing workflows.

**Why this priority**: The app is staff-facing and used live during events (including on-site on phones) — visual clarity and consistency directly affect operational speed and error rates.

**Independent Test**: Can be tested by walking through the existing standard flows (check-in, send email, catalog CRUD) on the redesigned UI and confirming the same tasks complete in the same or fewer steps, with no regression in responsive behaviour (375px / 768px / 1024px+).

**Acceptance Scenarios**:

1. **Given** a staff member on an existing view (e.g. Attendees, Check-in), **When** the redesigned UI ships, **Then** all existing actions remain reachable and functionally identical, with only visual/layout treatment changed.
2. **Given** a staff member on a mobile device, **When** they load any redesigned view, **Then** the view remains fully usable at ~375px with no horizontal scroll or broken touch targets.

---

### User Story 2 - Redesign stays within brand and accessibility constraints (Priority: P2)

Design/eng apply the redesign using the two-tier semantic token layer (primitive brand tokens + semantic roles per ADR-009) via CSS modules and the WCAG 2.2 rules already established for the app, rather than introducing a new component library or design language.

**Why this priority**: Prevents the redesign from silently regressing accessibility or security posture (e.g. reintroducing raw HTML injection risk) established in prior slices.

**Independent Test**: Can be tested by auditing redesigned views against the existing accessibility checklist (contrast, focus states, ARIA) and confirming no new UI framework (Tailwind/Shadcn/Tremor/Mantine) was introduced — self-hosted Manrope + Material Symbols subset fonts under `font-src 'self'` are the only permitted new front-end assets.

**Acceptance Scenarios**:

1. **Given** a redesigned view, **When** checked against WCAG 2.2 contrast and focus-visible rules, **Then** it passes with no regressions versus the current implementation.
2. **Given** the redesigned component set, **When** reviewed for dependencies, **Then** no new UI framework has been added outside CSS modules + semantic tokens (self-hosted fonts per ADR-009 excepted).

---

### User Story 3 - Redesign rolls out without disrupting in-flight vertical slices (Priority: P3)

The redesign initiative is delivered in a way that is compatible with the existing vertical-slice delivery model (ADR-004), so in-progress slice work is not blocked or forced into a big-bang rewrite.

**Why this priority**: Lower priority than not breaking staff workflows or accessibility, but still important for delivery risk — a big-bang UI rewrite conflicts with the project's incremental delivery approach.

**Independent Test**: Can be tested by confirming the redesign can be applied view-by-view (or shell-first) without requiring all in-progress slices to pause simultaneously.

**Acceptance Scenarios**:

1. **Given** an in-progress slice, **When** the redesign initiative begins, **Then** that slice's functional delivery is not blocked on the full redesign completing.

---

### User Story 4 - Staff choose a theme that persists across devices (Phase A) (Priority: P1)

Staff pick a visual theme (Aurora, Dark Aurora, and — if allowlisted — Celebration) that follows them across desktop ops and tablet check-in, without re-selecting per device.

**Why this priority**: Dark Aurora is needed for low-light in-room use and cross-device persistence is a stated staff need (desktop → tablet). This is Phase A (unblocked) foundational work every later surface depends on.

**Independent Test**: Sign in on one device, switch theme, sign in on a second device, confirm the same theme applies; confirm Celebration only appears for an allowlisted email and a stored `celebration` falls back to Aurora for a non-allowlisted signed-in user.

**Acceptance Scenarios**:

1. **Given** any signed-in staff member, **When** they open the theme switcher, **Then** Aurora and Dark Aurora are always offered, and Celebration is offered only if their email is on the Celebration allowlist.
2. **Given** a staff member who selected Dark Aurora on desktop, **When** they sign in on a tablet, **Then** Dark Aurora is applied automatically (cross-device backend preference).
3. **Given** a stored preference of `celebration` for a user no longer on the allowlist, **When** they load the app, **Then** the theme falls back to Aurora (server-side re-validation; stored preference not trusted).
4. **Given** the theme-preference write, **When** it is sent, **Then** it passes through the standard write-gate (validated, rate-limited, no PII).

---

### User Story 5 - Event-first navigation with standalone Events (Phase B) (Priority: P2)

Staff land on an Events overview and can reach any Event directly; a Program is an optional grouping/filter, not a mandatory drill-down. Standalone Events (no Program) are fully functional.

**Why this priority**: Core of the event-first redesign (ADR-008), but gated on ADR-007 feasibility (`X-REDESIGN-001`), so it sequences after Phase A.

**Independent Test**: Create a standalone Event (no `programId`) and confirm attendees, check-in, capacity, and campaigns all work; confirm selecting a Program filters/groups Events without being required to reach an Event.

**Acceptance Scenarios**:

1. **Given** the redesigned shell, **When** a staff member signs in, **Then** they land on an Events overview/list (not a Program-first drill-down).
2. **Given** a standalone Event with no `programId`, **When** a staff member opens it, **Then** Event Details, Attendees, Check-in, capacity, and Campaign all function normally.
3. **Given** a Program with several Events, **When** a staff member selects that Program, **Then** its Events are filtered/grouped — selecting the Program is not a precondition for opening an Event.
4. **Given** an Event, **When** its end date has passed, **Then** its status auto-derives to Completed; Active/Cancelled remain manual; publish state is tracked separately from status.

---

### User Story 6 - Registration and check-in via Contact↔Event associations (Phase B) (Priority: P2)

Attendance is modelled as a Contact↔Event association with labels (`registered` / `checked-in` / `customer` / `partner`). Staff check in, undo check-in, and remove attendees through the audited EMS write path; public/walk-in registration is written HubSpot-workflow-side.

**Why this priority**: Data-model migration that makes standalone Events coherent (ADR-007); gated on the same feasibility slots as US5.

**Independent Test**: Check in a registered attendee (label flips `registered` → `checked-in`, `checkedInAt` + scan method cached, audit recorded); undo reverses it; attempt to remove a `checked-in` attendee and confirm it is blocked; confirm EMS exposes no "register attendee" write.

**Acceptance Scenarios**:

1. **Given** a registered attendee, **When** staff check them in, **Then** the association label flips `registered` → `checked-in`, per-registration detail (`checkedInAt`, scan method, QR nonce) is written to Record Storage keyed by `contactId + eventId`, and the action is audited.
2. **Given** a checked-in attendee, **When** staff undo the check-in, **Then** the label reverts to `registered` and the action is audited.
3. **Given** a checked-in attendee, **When** staff attempt to remove them, **Then** removal is blocked (allowed only while `registered`).
4. **Given** the EMS UI, **When** staff look for a way to register a new attendee, **Then** none exists in EMS — registration is via the HubSpot form/workflow (public) or the embedded walk-in iframe (which registers only, never auto-checks-in).
5. **Given** an Event is archived, **When** the per-registration Record Storage cache is purged, **Then** the audit log remains the durable backstop for check-in history.

---

### Edge Cases

- What happens to a view that is mid-redesign when a slice needs to ship a functional change to that same view?
- How does the redesign handle views driven by `USE_MOCK_API` vs live data — should visual QA happen against both?
- What happens if a redesigned component's new markup breaks an existing Vitest render test that asserted on old DOM structure?
- How is a walk-in shown between form submission and the attendee appearing on the roster (propagation lag) — the UI must set this expectation rather than implying an instant add?
- What happens to existing `programs/{programId}/events/{eventId}/…` routes when navigation becomes event-first — the breaking contract change must update `api-contract.md` + `rbac.md` + `RouteGuard` together (`X-REDESIGN-003`).
- How do existing Program→Event catalog records behave after migration — they remain valid with a `programId`; migration adds standalone Events without forcing reparenting.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Phase A MUST preserve all existing functional behavior (navigation, data actions, RBAC-gated actions) in every view it touches — visual/token change only. Phase B MAY introduce the net-new information architecture and data model defined in [ADR-007](../../docs/decisions/007-hubspot-custom-objects-registration.md) / [ADR-008](../../docs/decisions/008-standalone-events-event-first-nav.md) (event-first navigation, standalone Events, registration-as-association); where it does, it MUST preserve existing RBAC/security boundaries and route all writes through the audited EMS path — no silent scope change outside those ADRs.
- **FR-002**: The redesign MUST implement theming via the two-tier semantic token layer (primitive brand tokens + a semantic role layer that themes remap) per [ADR-009](../../docs/decisions/009-redesign-ui-platform-theming-typography.md), extending the existing `--color-*` token convention (NOT the prototype's inline `var(--x)`-per-element pattern), and MUST NOT introduce Tailwind, Shadcn, Tremor, Mantine, or any other UI framework. Self-hosted Manrope + Material Symbols Outlined subset fonts (`font-src 'self'`) are permitted per ADR-009 and are not considered UI frameworks.
- **FR-003**: The redesign MUST meet or exceed current WCAG 2.2 compliance (contrast ≥ 4.5:1, focus-visible, keyboard nav, ARIA) for every view it touches; for the custom field pickers (FR-011), keyboard nav + ARIA + screen-reader support is a hard completion gate.
- **FR-004**: The redesign MUST remain responsive across phone (~375px), tablet (~768px), and desktop (≥1024px) for every view it touches.
- **FR-005**: The redesign MUST render all dynamic/remote data via JSX text interpolation only — no relaxation of the existing XSS-prevention rule.
- **FR-006**: Redesigned views MUST ship with updated Vitest render tests covering the new structure.
- **FR-007**: System MUST support incremental, foundation-first rollout: Phase A (semantic tokens, themes, fonts, pickers, theme preference) ships globally via tokens without rebuilding IA-dependent layouts; Phase B surfaces ship per-surface, consistent with the vertical-slice delivery model.
- **FR-008**: The redesign MUST provide three runtime themes selectable on `data-theme` — **Aurora** (default light, replacing today's default), **Celebration** (prototype pink `#EC6C93`, WCAG contrast-checked), and net-new **Dark Aurora** (dark) — with a user-chosen theme switcher. Celebration MUST appear in the switcher only for allowlisted emails (`CELEBRATION_THEME_EMAIL`); Aurora and Dark Aurora are selectable by everyone.
- **FR-009**: The system MUST persist the chosen theme as a per-user preference via a new write-gated backend endpoint (cross-device, validated, and rate-limited via Parameter `USER_PREFS_RATE_LIMIT_PER_HOUR`, default 60/user/hour). The preference MUST be keyed by the **Google account subject ID (non-PII)**, not the email address, so no PII is stored. Celebration MUST be re-validated server-side against the allowlist on load/login and MUST NOT be trusted from the stored preference — a stored `celebration` falls back to Aurora when the signed-in email is not allowlisted.
- **FR-010**: The redesign MUST self-host the Manrope subset and a Material Symbols Outlined subset (only used glyphs) bundled via Vite, adopting the prototype's size/weight scale, with CSP `font-src 'self'` and no Google Fonts / icon CDN.
- **FR-011**: The redesign MUST provide reusable custom field pickers (calendar / time / select popovers) matching all three themes (extending `CatalogPickerSelect`), replacing native date/time/select inputs for cross-theme consistency.
- **FR-012**: Phase B (event-first IA/routing, standalone Events, registration-as-association, live capacity/occupancy ±1, Campaign modal) MUST NOT ship until the `X-REDESIGN-001` feasibility gates pass (2 free HubSpot custom-object slots; workflows can set associations; ≤10 association labels per pairing; standard security write-gate).
- **FR-013**: The redesigned app shell MUST be **admin-only for now** (including the Overview dashboard) and MUST be role-aware by design so a future `check-in operator` role can be added without restructuring.
- **FR-014** *(Phase B)*: Navigation MUST be event-first — staff land on an Events overview and reach any Event directly; Program MUST be an optional grouping/filter, not a required drill-down. Every Event-scoped operation MUST function with `programId` absent (standalone Events).
- **FR-015** *(Phase B)*: Attendance MUST be modelled as a Contact↔Event association with labels `registered` / `checked-in` / `customer` / `partner`; check-in flips `registered` → `checked-in` (undo reverses). EMS's write surface is limited to check-in, undo check-in, remove attendee (blocked while `checked-in`), and catalog CRUD — EMS MUST NOT expose a public "register attendee" write.
- **FR-016** *(Phase B)*: Per-registration operational detail (`checkedInAt`, scan method, QR nonce/JWT) MUST live in Record Storage keyed by `contactId + eventId`, purged on Event archive, with the audit log as the durable backstop; all check-ins MUST flow through the audited EMS path (no HubSpot-only check-in writes).
- **FR-017** *(Phase B)*: Event status MUST follow Active / Cancelled (manual) + Completed (auto-derived when the end date passes); publish state MUST be tracked separately from status.
- **FR-018** *(Phase B)*: Event-first routing MUST update `docs/api-contract.md`, `docs/rbac.md`, and `RouteGuard` together for the breaking `programs/{programId}/events/{eventId}/…` → event-scoped change (`X-REDESIGN-003`); no HubSpot property/schema names may be used until verified in `docs/hubspot-schema.md` (`X-REDESIGN-004`).
- **FR-019** *(Phase B)*: Copy fixes MUST be applied per grilling outcomes — remove "auto-checks in on scan" wording; QR summary fixed set = name, company, email, account manager, attendee type, current status; the walk-in UI MUST communicate roster propagation lag; rename the prototype's "HubSpot list" to "segment".

### Key Entities

- **View**: An EMS screen (existing or net-new, e.g. Overview, Programs & Events, Event Details, Attendees, Check-in, Email/Campaign, Audit Log) that the redesign delivers or restyles.
- **Design Token**: A primitive brand token or a semantic role token in the two-tier token layer that themes remap; components reference semantic tokens only.
- **Theme**: A named set of semantic-token values (`aurora` / `celebration` / `darkAurora`) applied via `data-theme`.
- **Theme Preference**: A per-user, cross-device stored choice of theme (backend), with Celebration re-validated server-side against the allowlist.
- **Program / Event (custom objects)** *(Phase B)*: HubSpot custom objects per ADR-007; Program is optional and Event is the primary entity (ADR-008).
- **Registration** *(Phase B)*: a Contact↔Event association with labels (`registered` / `checked-in` / `customer` / `partner`) per ADR-007, with per-registration detail in Record Storage.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of redesigned views pass the existing accessibility and responsive checklists with zero regressions versus their pre-redesign state.
- **SC-002**: Zero new UI framework dependencies (Tailwind/Shadcn/Tremor/Mantine/etc.) are added as a result of the redesign; self-hosted Manrope + Material Symbols subset fonts (per ADR-009) are the only new front-end assets, served under `font-src 'self'`.
- **SC-003**: All existing standard-flow checklists (check-in, send email, catalog CRUD, archive/cancel) complete in the same or fewer clicks on the redesigned UI as on the current UI.
- **SC-004**: All three themes (Aurora, Celebration, Dark Aurora) pass the WCAG 2.2 contrast audit (`docs/ui-a11y-audit.md`) on interactive/label text, and the custom field pickers are fully operable by keyboard and screen reader (completion gate).
- **SC-005**: Theme preference persists across devices for a signed-in user, and a stored `celebration` preference resolves to Aurora for any non-allowlisted signed-in email (server-side re-validation) in 100% of cases.
- **SC-006** *(Phase B)*: A standalone Event (no `programId`) supports attendees, check-in, capacity, and campaigns with zero Program dependency; check-in / undo / remove-attendee all produce an audit entry, and no EMS "register attendee" write path exists.

## Assumptions

- The redesign initiative is both a visual/UX refresh **and** a data-model migration: Phase A is presentation-layer (theming, typography, pickers) plus one small backend theme-preference endpoint; Phase B (gated) introduces the HubSpot custom-objects data model and event-first IA per ADR-007/008. It is not a rewrite of unrelated backend systems.
- The authoritative visual source of truth is the `Frontend/design_handoff 2/` prototype (see Clarifications 2026-07-13); redesigned views are recreated to match it. Authoritative decisions live in ADR-007/008/009 and CONTEXT.md § *Redesign transition — target model*.
- The brand primitive palette is retained and extended into a semantic token layer; the three themes (Aurora default, Celebration pink `#EC6C93`, net-new Dark Aurora) remap semantic tokens per ADR-009.
- Delivery is foundation-first: Phase A ships globally via tokens; Phase B ships per-surface once feasibility gates pass — consistent with the vertical-slice delivery model (ADR-004).
- Security and RBAC boundaries are preserved and extended: the app shell is admin-only for now and role-aware by design (FR-013); Phase B writes go through the audited EMS write path.
- No dedicated staff-feedback metric/channel is tracked for this initiative in v1; success is measured structurally (accessibility/responsive parity, no new UI-framework dependencies, flow parity) rather than via a usability survey.

## Scope

This initiative covers the **entire Frontend application** plus supporting backend work, delivered in two phases:

- **Phase A (unblocked):** the visual foundation — semantic token layer + 3 themes (incl. net-new Dark Aurora) + user switcher, self-hosted Manrope + Material Symbols, shared a11y field pickers, and a write-gated backend theme-preference endpoint. Applied globally via tokens without rebuilding IA-dependent layouts.
- **Phase B (blocked on `X-REDESIGN-001` feasibility gates):** event-first IA/routing, standalone Events, registration-as-association (ADR-007/008), live capacity/occupancy ±1, and the Campaign modal. Nothing in Phase B ships until the feasibility gates pass.

Out of scope: campaign drafts (`FE-REDESIGN-008`, parked); any HubSpot property/schema names not verified in `docs/hubspot-schema.md`.
